# REVIEW — 2026-07-21 (post-cancellation rebuild; anchor: 9.75 / 9.8)

## 1. Headline

The user pulled the submission on 07-16 because real-device testing surfaced
what the headless rig couldn't: an information architecture that reviewers
called "too complicated," meals that appeared and vanished on their own, and
saved plans that didn't match what was built. Five days later the app has a
different spine: **five tabs with one noun each** (Explore · Events · Map ·
Planner · My Plans), a Planner that shows **only what the user added** (meals
are opt-in via "🍴 Add a restaurant"; suggestions are pinned by id and never
change underneath you), events as **first-class schedule cards** that work
without dates, saved plans that replay the **exact card order** with events,
notes, and honest per-person pricing (admissions counted, "+ ticket" when a
show is in the day), a Food map layer that grew 16 → 64 pins, and a PDF
export that works in the native shell. Every fix in this cycle came from the
owner's own device testing, and every one was verified headless before
handing back. **Maya 9.75 → 9.8, Tom & Rachel 9.8 → 9.85. Verdict: GO after
the store metadata is refreshed** — the app outran its own screenshots.

Justifications (anti-inflation: every move names shipped behavior):

- **Clarity (+0.25 both):** the single biggest strategic fix of the cycle.
  "My Trip" — one page doing workbench + archive + share — became **Planner**
  (auto-saving workbench) and **My Plans** (shelf), each a nav tab, with one
  verb family ("+ Add to Planner" → "Save copy" → My Plans). And the planner
  stopped volunteering content: no auto-suggested restaurants (opt-in
  button), no phantom meals riding along with stops. The reviewer complaint
  that triggered the cancellation ("which is the main point of my app?") has
  a structural answer now, not a wording patch.
- **Navigation (+0.05 both):** the last known dead end died — "+ Add to
  Planner" now *always* produces a visible card, even for a place previously
  ✕'d out of the plan (explicit-add-wins in toggleSave). Post-save, a
  "Start fresh" banner resets to a true first-open page (verified down to
  the sample-weekend button working, which the reset used to silently break).
  Removing the last card auto-clears the leftover day frames.
- **Trust (Maya +0.1, T&R +0.15):** every count on every surface now derives
  from the *rendered cards* (header "2 stops" = the 2 cards below it; "N
  meals picked" inventory noise deleted); suggestions are pinned so removing
  a stop can never silently swap a restaurant (verified: Caravaggio +
  J.G. Melon identical across two removals); saved plans are WYSIWYG (exact
  order replay incl. events, ✎ notes, map chips); price estimates stopped
  lying by omission — museum admissions count ("Adults $25 · Students $15" →
  $25), free is $0, unknown stays out, events append "+ ticket". Visitors
  gain more because the saved plan and its PDF are *their* artifact on the
  sidewalk.
- **First impression (Maya +0.05):** the empty Planner is one coherent state
  (map emoji, one sentence, one button) regardless of how you arrived —
  first install, Start fresh, or removing everything.
- **Content, Delight (flat):** dataset unchanged this cycle; the Food layer
  64-pin fix is exposure of existing content, not new content — it shows up
  in Navigation/Trust, not Content.

## 5. What's working — keep this

The invitation-first grammar. Every surprise the app used to spring —
auto-meals, anchor-hopping suggestions, events quarantined on a shelf, dead
add-buttons — is now either the user's explicit act or one labeled tap away.
That single principle ("only what I added, everything one tap away") resolved
more reviewer confusion than any copy change of the previous three cycles.
